## 공고에서 읽은 핵심 과제

채팅팀은 당근의 여러 서비스가 만나는 교차로를 더 편리하고 유연하게 만드는 조직이다. 공고상 현재 과제는 크게 세 가지다.

   - 다양한 서비스 요구를 담을 수 있는 확장 가능한 메시지 구조
   - 일관된 화면 흐름

3. 채팅의 기본 역량 확장
   - 웹 SDK 초기 구축
   - 웹 어드민 고도화

## 채팅팀 블로그에서 읽은 방향성

자료: https://about.daangn.com/blog/archive/%EB%8B%B9%EA%B7%BC-%EC%B1%84%ED%8C%85%ED%8C%80-%EC%B1%84%EC%9A%A9-%EB%B9%84%EC%A0%84-%EB%AC%B8%ED%99%94/

- 당근 채팅은 단순 메신저가 아니라 중고거래, 모임, 알바, 부동산 등 여러 서비스의 목적이 이어지는 접점이다.
- 채팅팀은 End User뿐 아니라 채팅 퍼널을 사용하는 내부 서비스 팀도 고객으로 본다.
- 반복되는 요구를 공통화하고, 서비스별 맥락을 해치지 않도록 유연한 채팅 구조를 만든다.
- 예시 과제:
  - 거래/모임/아파트 등 서비스 맥락별 채팅 리스트 뷰 타입 개선
  - 서비스별 시스템 메시지를 커스텀하면서도 톤은 일관되게 유지하는 시스템
  - 당근전화처럼 빠르게 PoC하고 지표까지 검증하는 실행 방식

## carrot-chat으로 보여줄 메시지

`carrot-chat`은 공고의 핵심 과제를 작게 재현한 포트폴리오다. 제출 시에는 "채팅 앱을 만들었다"보다 아래 메시지를 강조한다.

- UI보다 먼저 재사용 가능한 헤드리스 SDK 경계를 잡았다.
- React 바인딩은 SDK를 소비하는 얇은 레이어로 두어, 나중에 웹뷰/어드민/샘플 앱으로 확장할 수 있게 했다.
- WebSocket 연결, 자동 재연결, 하트비트, 이벤트 팬아웃을 SDK 내부로 모았다.
- 서버측 정책 제어(연락처 마스킹)를 Durable Object에서 강제해 클라이언트 우회 가능성을 줄였다.
- 테스트 전략을 순수 함수/가짜 WebSocket/컴포넌트/E2E로 나누어 안전하게 검증했다.
- 접근성을 위해 대화 영역에 `role="log"`, `aria-live`, 입력 라벨을 적용했다.

## 제출 전 위험 요소

- 실제 당근 내부 SDK와 동일하다고 말하면 안 된다. "공고에서 읽은 과제를 작게 재현했다"라고 표현한다.
- `branches` 폴더의 결과물은 사용자가 랜딩 페이지만 작업했다고 했으므로 핵심 포트폴리오 근거에서 제외한다.
- 다른 프로젝트에서 확인한 Cloudflare 계정/토큰/운영 세부값은 지원서나 공개 문서에 쓰지 않는다.
- 성과 수치가 로컬 문서로 확인되지 않는 프로젝트는 정량 수치를 임의로 만들지 않는다. 수치가 필요하면 사용자가 제공해야 한다.
