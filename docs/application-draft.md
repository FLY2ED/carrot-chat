# Application Draft

당근 `Software Engineer, Frontend (인턴) | 채팅` 지원용 초안입니다.

주의:

- 아래 문장은 그대로 제출 가능한 톤으로 작성했지만, 학교/학년/근무 가능 여부/정확한 프로젝트 기간/성과 수치는 사용자가 직접 확인해 넣어야 합니다.
- 정량 성과를 임의로 만들지 않았습니다.
- `branches` 폴더 프로젝트는 핵심 근거에서 제외했습니다.

## 1. 한 줄 소개

React와 TypeScript로 사용자 경험을 빠르게 구현하되, SDK 경계와 테스트 전략까지 함께 설계하는 프론트엔드 개발자 박성재입니다.

## 2. 포트폴리오 한 줄 설명

당근 채팅팀의 웹 SDK, 웹채팅, 운영 정책 제어 과제를 작게 재현하기 위해 React + TypeScript 프론트엔드와 Cloudflare Durable Objects(WebSocket) 기반 실시간 채팅 SDK 데모를 만들었습니다.

## 3. 지원동기

당근 채팅팀에 지원한 이유는 채팅이 단순한 메시지 입력창이 아니라 당근의 여러 서비스가 만나는 접점이라는 점 때문입니다. 중고거래, 모임, 알바, 부동산처럼 목적이 다른 서비스에서 시작된 연결이 채팅 안에서 끊기지 않고 완결되려면, 화면을 잘 만드는 것뿐 아니라 서비스별 정책, 메시지 구조, 운영 도구, 테스트 인프라까지 함께 설계해야 한다고 이해했습니다.

저는 이 공고를 보고 `carrot-chat`이라는 작은 웹채팅 SDK 데모를 만들었습니다. UI를 먼저 붙이는 대신, WebSocket 연결과 자동 재연결, 하트비트, 이벤트 팬아웃을 담당하는 프레임워크 비종속 `chat-core`를 분리했습니다. React는 이 SDK를 소비하는 바인딩으로 두고, 상태 관리는 Zustand store로 연결했습니다. 서버는 Cloudflare Durable Objects를 사용해 방 단위 연결과 메시지 히스토리, presence, typing, read receipt, 연락처 마스킹 정책을 관리하도록 구성했습니다.

이 과정에서 당근 채팅팀이 말하는 "여러 서비스가 동일한 품질의 채팅 기능을 쉽게 사용할 수 있도록 표준과 시스템을 구축한다"는 일이 어떤 고민을 요구하는지 작게나마 경험했습니다. 인턴으로 합류한다면 먼저 기존 코드와 팀의 맥락을 빠르게 읽고, 테스트 가능한 작은 단위로 웹채팅 경험과 운영 도구를 개선하는 데 기여하고 싶습니다.

## 4. 대표 프로젝트: carrot-chat

`carrot-chat`은 재사용 가능한 웹채팅 SDK를 목표로 만든 데모입니다. React + TypeScript UI와 Cloudflare Durable Objects 기반 WebSocket 백엔드를 연결해 두 개의 독립 클라이언트가 같은 방에서 실시간 메시지, 읽음 상태, 타이핑 상태, 접속자 수를 동기화하도록 만들었습니다.

가장 중요하게 둔 설계 원칙은 UI와 통신 로직의 분리였습니다. `src/chat-core`는 React나 DOM을 모르는 헤드리스 SDK로 만들고, `src/react`는 이 SDK를 Zustand store와 컴포넌트에 연결하는 바인딩으로 두었습니다. 덕분에 WebSocket 연결 상태, 재연결, 하트비트, 서버 이벤트 파싱을 한곳에서 다룰 수 있었고, React 컴포넌트는 메시지 목록, 입력창, 상태 배지만 책임지게 만들 수 있었습니다.

서버에서는 Durable Object를 방 단위 단일 진실 공급원으로 사용했습니다. 연결된 WebSocket을 통해 메시지를 브로드캐스트하고, SQLite 히스토리로 최근 메시지를 복구하며, 전화번호/이메일 같은 연락처는 서버측에서 마스킹했습니다. 이 정책은 사용자가 클라이언트를 수정하더라도 우회하기 어렵게 서버에서 강제해야 한다고 판단했습니다.

테스트는 위험이 큰 부분부터 나누어 작성했습니다. 연락처 마스킹과 재연결 백오프는 순수 함수 단위 테스트로 빠르게 검증했고, `ChatClient`는 가짜 WebSocket과 fake timer로 연결 상태머신을 검증했습니다. 입력창은 Testing Library로 전송/타이핑 동작을 확인했고, Playwright E2E로 실제 Durable Object 런타임에서 두 클라이언트 간 메시지 동기화와 서버측 마스킹을 검증했습니다.

현재 검증 결과는 `typecheck`, Vitest 14개 테스트, production build, Playwright E2E 2개 테스트 모두 통과입니다.

## 5. 문제 해결 경험: 실시간 채팅에서 신뢰할 수 있는 연결 만들기

실시간 채팅 데모에서 가장 먼저 해결하고 싶었던 문제는 "일시적인 네트워크 끊김을 사용자가 바로 장애로 느끼지 않게 하는 것"이었습니다. 채팅 UI만 구현하면 정상 연결 상황에서는 동작하지만, 실제 채팅 경험에서는 연결이 끊기고 다시 붙는 순간의 상태 관리가 중요합니다.

그래서 `ChatClient`를 별도 클래스로 분리하고, `connecting`, `open`, `reconnecting`, `closed` 상태를 명시적으로 관리했습니다. 비정상 종료가 발생하면 지수 백오프와 지터를 적용해 재연결하고, 의도적으로 닫은 경우에는 다시 연결하지 않도록 구분했습니다. 하트비트는 SDK 내부에서 처리해 React 컴포넌트가 연결 유지 로직을 알 필요 없게 했습니다.

이 로직은 UI에서 수동으로 확인하기보다 테스트 가능한 구조로 만들었습니다. fake WebSocket과 fake timer를 주입해 연결 성공, 비정상 종료 후 재연결, 의도적 종료, heartbeat ping 전송을 단위 테스트로 검증했습니다. 이 경험을 통해 사용자에게 보이는 기능보다 먼저 상태 전이를 작게 모델링하고 검증하는 방식이 실시간 UI의 안정성을 높인다는 것을 배웠습니다.

## 6. 테스트 전략 경험

저는 테스트를 "많이 작성하는 것"보다 "실패했을 때 원인을 빨리 좁힐 수 있게 나누는 것"으로 보고 있습니다. `carrot-chat`에서는 네 가지 층으로 테스트를 나눴습니다.

첫째, 연락처 마스킹과 재연결 백오프처럼 입출력이 명확한 로직은 순수 함수 테스트로 검증했습니다. 둘째, WebSocket 연결 상태머신은 가짜 WebSocket과 fake timer를 이용해 브라우저나 서버 없이 검증했습니다. 셋째, 입력창 컴포넌트는 Testing Library로 사용자의 입력, 전송, 초기화, 타이핑 이벤트를 확인했습니다. 넷째, 실제 런타임 통합은 Playwright로 확인했습니다. 같은 방에 접속한 두 클라이언트가 메시지와 읽음 상태를 실시간으로 동기화하는지, 연락처가 서버에서 마스킹되는지 검증했습니다.

이 구조 덕분에 빠른 피드백과 실제 환경 검증을 분리할 수 있었습니다. 당근 채팅팀에서 SDK를 안전하게 배포하기 위한 테스트 인프라를 다룬다면, 우선 변경이 잦은 이벤트 모델과 정책 로직은 빠른 테스트로 보호하고, 브라우저/WebSocket/운영 콘솔 플로우는 E2E로 보완하는 방식으로 기여하고 싶습니다.

## 7. 협업과 오너십 경험

여러 프로젝트를 진행하며 제가 중요하게 생각하게 된 것은 "구현 결과만 남기는 것보다 의사결정의 이유와 운영상의 주의점을 같이 남기는 것"입니다. 예를 들어 `artdata` 프로젝트에서는 입시 분석, 리포트, 정책, AI 추출 로직처럼 요구사항이 복잡한 기능을 다루며 QA 문서와 정책 문서를 함께 남겼습니다. `Monggeul`과 `Dangaro`에서는 프론트엔드뿐 아니라 API, 인증, 배포, 운영 체크리스트까지 함께 보며 문제를 끝까지 닫는 경험을 했습니다. `landing-security` 계열에서는 개인정보 마스킹, 암호화, 접근 로그, IP 제한처럼 운영 정책이 코드로 강제되어야 하는 영역을 공통 모듈로 정리했습니다.

이 경험들은 모두 채팅팀의 업무와 연결된다고 생각합니다. 채팅은 한 팀의 화면이 아니라 여러 서비스 팀과 내부 운영자가 함께 쓰는 기반 기능이기 때문에, 기능을 만드는 것만큼 정책, 로그, 테스트, 문서가 중요합니다. 저는 새로운 코드베이스에 들어가면 먼저 구조와 기존 스타일을 읽고, 변경 이유를 작게 기록하면서, 다음 사람이 같은 맥락을 빠르게 이해할 수 있게 하는 방식으로 일하겠습니다.

특히 `artdata` 실서비스에서는 학생↔강사 **Socket.io 기반 실시간 1:1 채팅을 원작자로 설계·구현**했습니다. 룸 기반 WebSocket 인증, 읽음 확인, 타이핑 인디케이터, 파일 메시지, 어드민 채팅 모니터링, 그리고 **연락처/이메일을 서버에서 자동 마스킹해 플랫폼 외부 거래를 차단하는 정책 제어**까지 포함했습니다. 토큰 만료 시 소켓을 자동 재인증·재연결하는 하드닝도 후속으로 더했습니다. 이 부분이 당근 채팅팀의 도메인과 가장 가까운 경험이라고 생각합니다.

또한 `KORI VOCA`(React Native + TypeScript 한국어 학습 앱, App Store 출시, ST창업오디션 최우수상)와 `Desktop Dday`(Electron + Vue 바탕화면 위젯, Microsoft·Google·Apple 3개 스토어 출시, Disquiet 위클리 프로덕트 선정, 운영 후 매각)처럼 작은 제품도 출시·운영·종료까지 책임지고 닫아본 경험이 있습니다. 정량 지표와 유저 보이스 둘 다 살피며 작게 실험하고 빠르게 다듬는 일에 익숙합니다.

## 8. 당근 채팅팀에서 기여하고 싶은 일

합류한다면 먼저 웹채팅과 운영 관리 콘솔의 작은 개선 과제부터 빠르게 맡아 팀의 코드 스타일과 배포 흐름을 익히고 싶습니다. 특히 세 영역에서 기여하고 싶습니다.

첫째, SDK와 React UI 사이의 경계를 명확히 유지하며 재사용 가능한 컴포넌트와 이벤트 모델을 만드는 일입니다. 둘째, 운영자가 채팅방 정책이나 메시지 상태를 이해하기 쉽게 만드는 어드민 화면 개선입니다. 셋째, 안전한 배포를 위한 테스트 보강입니다. 단위 테스트로 보호할 수 있는 정책/상태 로직과 E2E로 확인해야 하는 브라우저/WebSocket 흐름을 구분해, 변경 속도와 안정성을 함께 가져가는 데 기여하고 싶습니다.

무엇보다 인턴으로서 가장 먼저 할 일은 기존 코드와 팀의 문제 정의 방식을 빠르게 배우는 것이라고 생각합니다. 제가 만든 데모는 작은 재현일 뿐이지만, 이 경험을 바탕으로 실제 서비스에서는 더 복잡한 제약과 사용자 맥락을 겸손하게 읽고, 작고 검증 가능한 개선을 꾸준히 쌓겠습니다.

## 10. 짧은 자기소개 버전

React와 TypeScript로 사용자 인터페이스를 구현하면서도, 재사용 가능한 로직 경계와 테스트 전략을 함께 고민하는 프론트엔드 개발자입니다. 이번 지원을 위해 `carrot-chat`이라는 웹채팅 SDK 데모를 만들었고, WebSocket 연결/재연결/하트비트/이벤트 모델을 프레임워크 비종속 core로 분리했습니다. React는 Zustand store와 컴포넌트를 통해 이 SDK를 소비하도록 구성했고, 서버는 Cloudflare Durable Objects로 방 단위 실시간 동기화와 정책 제어를 담당하게 했습니다. 빠르게 구현하는 것에서 끝내지 않고, 순수 함수/상태머신/컴포넌트/E2E 테스트를 나누어 검증했습니다.
